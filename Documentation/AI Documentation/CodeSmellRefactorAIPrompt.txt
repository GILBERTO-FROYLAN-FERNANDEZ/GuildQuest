## 1. **Static Abuse**
Nearly all methods and fields are static (`page`, `allUsers`, `getUser()`, `login()`, `displayOptions()`, etc.). This creates tight coupling, makes unit testing nearly impossible, and prevents proper object-oriented design.

## 2. **God Class / Violation of Single Responsibility Principle**
UserUI handles too many responsibilities: user authentication, campaign creation/editing/viewing, character management, settings, and UI control flow. This should be split into multiple focused classes.

## 3. **Mutable Global State**
The static `allUsers` HashMap is shared mutable state accessible throughout the application, leading to potential concurrency issues and difficult-to-track bugs.

## 4. **Poor Separation of Concerns**
UI presentation logic, business logic, and control flow are all mixed together. Methods directly call `System.out.println()`, manipulate domain objects, and control navigation flow.

## 5. **Magic Numbers**
Hard-coded integers (1, 2, 3, 4, 5, 0) throughout switch statements with no named constants, making the code harder to maintain and understand.

## 6. **Null as Control Flow**
Using `null` returns to indicate "exit" or "go back" (e.g., in `login()`, `accessUser()`) is an anti-pattern that obscures intent and can lead to NullPointerExceptions.

## 7. **Inappropriate Error Handling**
`throw new Error("Invalid Choice, should not be possible")` for unreachable code suggests defensive programming mixed with poor validation design.

## 8. **Long Methods with Complex Control Flow**
Methods like `getUser()` and `displayOptions()` contain nested loops, multiple conditionals, and switch statements that make them difficult to understand and test.

## For Magic Numbers in Switch Statements

### **Replace Magic Number with Symbolic Constant (Enum)**
Create an enum to represent menu choices instead of using raw integers (1, 2, 3, 4, 5, 0). This gives semantic meaning to values and prevents confusion.

**Benefits:**
- Self-documenting code (e.g., `MenuOption.CREATE_CAMPAIGN` vs `1`)
- Type safety - compiler catches invalid options
- Easier to maintain - adding/removing options doesn't require hunting for all magic numbers
- Clear intent at call sites

**Example concept:** Define an enum like `MenuOption` with values (CREATE_CAMPAIGN, EDIT_CAMPAIGN, VIEW_CAMPAIGN, CHARACTER_MANAGER, SETTINGS, EXIT), each associated with its numeric value. The switch statement then operates on the enum type.

## For the Long Method `displayOptions`

### **Replace Conditional with Polymorphism / Command Pattern**
Create a command object for each menu action instead of switch cases. Store commands in a map keyed by the choice number. When a choice is made, look up and execute the corresponding command.

**Benefits:**
- Eliminates the switch statement entirely
- Each action becomes its own class/object with a single responsibility
- Easy to add new menu options without modifying existing code (Open/Closed Principle)
- Commands can be tested in isolation

**Concept:** Each menu action (createCampaign, editCampaign, etc.) becomes a command object implementing a common interface with an `execute(User)` method.

### **Introduce Explaining Variable**
Break down the complex prompt string construction into smaller, named variables that explain what each section represents.

**Benefits:**
- Makes the intention of string building clearer
- Easier to modify individual sections
- Reduces cognitive load when reading the method

### **Replace Loop with Collection Pipeline**
If the menu system becomes more complex, convert the while loop into a more declarative structure where menu state is explicit rather than controlled by a loop variable.

**Benefits:**
- More explicit control flow
- Easier to reason about state transitions
- Reduces mutation and side effects
